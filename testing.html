üîç Detailed Test Cases

Test Case 1: Empty Selection Validation

  1. Load the page
  2. Click "Validate Selection" button
  3. Try to submit the form
Expected: Error messages should appear under both dropdowns indicating they are required. Submit button should remain disabled. Validation rules should show pending/invalid states.

Test Case 2: Same Model Selection

  1. Select "GPT-4" in Model A dropdown
  2. Select "GPT-4" in Model B dropdown
  3. Observe validation feedback
Expected: Both dropdowns should show red border and error icon. Error message should indicate models cannot be the same. Submit button should be disabled.

Test Case 3: Valid Different Selection

  1. Select "GPT-4" in Model A dropdown
  2. Select "Claude 3 Opus" in Model B dropdown
  3. Observe validation feedback
Expected: Both dropdowns should show green border and checkmark icon. Validation status should show success message. Submit button should be enabled. All validation rules should show green checkmarks.

Test Case 4: Real-time Validation

  1. Select "GPT-4" in Model A
  2. Select "Claude 3 Opus" in Model B (valid state)
  3. Change Model B to "GPT-4" (invalid state)
  4. Change Model B back to "Claude 3 Opus" (valid again)
Expected: Validation should update immediately on each change without clicking any buttons. Visual indicators should change in real-time.

Test Case 5: Form Submission

  1. Set up a valid selection (different models)
  2. Click "Compare Models" button
  3. Try submitting invalid form (same models or empty)
Expected: Valid submission should show success alert with model names. Invalid submission should show validation errors and form shake animation.

Test Case 6: Reset Functionality

  1. Make any selections in both dropdowns
  2. Click "Reset Selection" button
  3. Verify all fields are cleared
Expected: All dropdowns reset to "Select..." state. All validation states reset. Form shows shake animation. Summary section shows "Not selected" for both models.

Test Case 7: Responsive Design

  1. Resize browser window to mobile size (< 768px width)
  2. Test all functionality on mobile layout
  3. Verify dropdowns are usable on touch devices
Expected: Layout should stack vertically on mobile. All functionality should work the same. Buttons should be full-width for better touch interaction.

Test Case 8: Accessibility

  1. Navigate using only keyboard (Tab, Enter, Arrow keys)
  2. Test with screen reader if available
  3. Verify focus indicators are visible
Expected: All interactive elements should be keyboard accessible. Focus indicators should be clearly visible. Labels should be properly associated with form controls.

üéØ Key Features to Verify

  • Validation Rules:
    • Both dropdowns must have a selection (not empty)
    • Selected models must be different
    • Form only submits when both rules are satisfied
  • User Experience:
    • Real-time validation feedback as user types/selects
    • Clear error messages and visual indicators
    • Success states with green checkmarks
    • Summary section showing current selections
    • Animated feedback (shake on reset/error)
  • Technical Implementation:
    • No JavaScript errors in console
    • Form prevents submission when invalid
    • Proper event handling for all interactions
    • Clean state management and UI updates
